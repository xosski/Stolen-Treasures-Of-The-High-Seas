What your screenshot suggests is:

the app handled force as an invalid subcommand,
but somewhere in that error path it still tried to run systeminfo,
and because you were in Git Bash / MINGW64, systeminfo was not found in that shell context.

That matters because if an app launches systeminfo by bare name instead of an absolute path, it is relying on PATH lookup / shell resolution. When software does that, the risk is not “the word force is magic,” but that a spawned helper command can be intercepted or replaced if command lookup is sloppy.

The defensive takeaway is:

possible risk: command/path hijacking
not automatically a vulnerability: only if the app shells out unsafely or trusts PATH in a risky way
stronger risk if: it uses a shell call instead of direct process exec, or does not pin the full path to the Windows binary
weaker risk if: it calls the binary directly by absolute path and validates inputs

So the interesting signal here is not really the invalid force command itself. It is that the app appears to invoke systeminfo during handling, probably for environment detection or diagnostics. If that invocation is done lazily, then yes, that could be a hook point.
